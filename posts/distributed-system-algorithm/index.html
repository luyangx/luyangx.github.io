分布式系统中的一些一致性协议和算法

November 15, 2016

分布式系统中涉及到的一些一致性的协议和算法, 简单记录下.

2PC(Two-Phase Commit) 二阶段提交

将事务提交分成2个阶段

阶段一: 提交事务请求 (投票阶段)

  1. 事务询问

    协调者向参与者发送事务内容, 询问是否可以执行事务提交操作, 并开始等待参与者的响应.

  2. 执行事务

    参与者节点执行事务操作, 并将 Undo 和 Redo 信息记入事务日志中.

  3. 参与者向协调者反馈事务询问响应

    如果参与者执行了事务操作, 反馈给协调者: Yes, 表示事务可以提交; 否则, 反馈: No, 表示事务不可以执行.

阶段二: 执行事务提交

协调者会根据参与者的反馈情况决定最终是否可以进行事务提交操作, 包括2种可能:

  • 执行事务提交

  1. 发送提交请求

    协调者向参与者节点发出 Commit 请求.

  2. 事务提交

    参与者收到 Commit 请求后, 进行事务提交操作, 之后释放调整个过程中占用的资源.

  3. 反馈事务提交结果

    参与者在完成事务提交后, 向协助者发送 Ack 信息.

  4. 完成事务

    协调者接收到参与者反馈的 Ack 后, 完成事务.

  • 中断事务

  1. 发送回滚请求

    协调者向参与者节点发出 Rollback 请求

  2. 事务回滚

    参与者收到 Rollback 请求后, 利用阶段一中记录的 Undo 信息进行事务回滚操作, 之后释放调整个过程中占用的资源.

  3. 反馈事务回滚结果

    参与者在完成事务回滚后, 向协助者发送 Ack 信息.

  4. 中断事务

    协调者接收到参与者反馈的 Ack 后, 完成事务中断.

优点: 原理简单, 实现方便

缺点:

  • 同步阻塞

    参与者与事务操作的逻辑是处于阻塞状态, 在等待其他参与者响应过程中, 无法进行其他任何操作

  • 单点问题

    第二阶段, 参与者如果出现问题, 将无法完成事务操作.

  • 数据不一致

    第二阶段, 由于各种原因, 只有部分参与者收到 Commit 请求, 导致部分参与者提交了事务, 另一部分没有提交事务, 造成数据不一致.

3PC(Three-Phase Commit) 三阶段提交

在二阶段提交中增加了一个 PreCommit(预提交) 阶段

阶段一: CanCommit

  1. 事务询问

    协调者向参与者发送一个包含事务内容的 canCommit 请求, 询问是否可以执行事务提交操作, 并开始等待参与者的响应.

  2. 参与者向协调者反馈事务询问的响应

    参与者收到 canCommit 请求后, 正常情况下, 如果可以顺利执行事务, 反馈: Yes, 并进入预备状态, 否则反馈: No.

阶段二: PreCommit

协调者会根据参与者的反馈情况决定是否可以进行事务 PreCommit 操作, 包括2种可能:

  • 执行事务预提交

  1. 发送预提交请求

    协调者向参与者节点发出 PreCommit 请求, 并进入 Prepared 阶段.

  2. 事务预提交

    参与者收到 PreCommit 请求后, 会执行事务操纵, 并将 Undo 和 Redo 信息记录到事务日志中.

  3. 参与者向协调者反馈事务执行的响应

    如果参与者成功执行了事务操作, 反馈: Yes, 同时等待最终的指令: 提交(Commit)和终止(Abort).

  • 中断事务

任何一个参与者反馈: No, 或等待响应超时, 那么协调者会中断事务.

  1. 发送中断请求

    协调者向参与者发送 Abort 请求

  2. 中断事务

    参与者收到 Abort 请求或等待协调者请求超时后, 都会中断事务.

阶段三: 执行事务提交

该阶段将进行事务的真正提交, 包括2种可能:

  • 执行事务提交

  1. 发送提交请求

    协调者收到所有参与者得 Ack, 那么它将从 “预提交” 状态转换到 “提交” 状态, 并向参与者发送 doCommit 请求.

  2. 事务提交

    参与者收到 doCommit 请求后, 进行事务真正提交操作, 之后释放调整个过程中占用的资源.

  3. 反馈事务提交结果

    参与者在完成事务提交后, 向协调者发送 Ack 信息.

  4. 完成事务

    协调者接收到参与者反馈的 Ack 后, 完成事务.

  • 中断事务

任意一个参悟着反馈: No, 或等待参与者响应超时, 中断事务

  1. 发送中断请求

    协调者向参与者节点发出 Abort 请求

  2. 事务回滚

    参与者收到 Rollback 请求后, 利用阶段二中记录的 Undo 信息进行事务回滚操作, 之后释放调整个过程中占用的资源.

  3. 反馈事务回滚结果

    参与者在完成事务回滚后, 向协助者发送 Ack 信息.

  4. 中断事务

    协调者接收到参与者反馈的 Ack 后, 完成事务中断.

阶段三可能会存在2中故障:

  • 协调者出现问题
  • 协调者和参与者直接网络出现故障

无论哪种故障, 最终都会导致参与者无法及时手动协调者的 doCommit 或是 Abort 请求, 针对这样的异常情况, 参与者都会在等待超时后, 继续进行事务提交.

优点: 相对于二阶段提交, 降低了参与者的阻塞范围, 并能在单点故障后继续达成一致

缺点: 参与者收到 preCommit 请求后, 网络出现分区, 会造成数据不一致
