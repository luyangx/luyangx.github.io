TCP . 重传机制/滑动窗口/拥塞处理

April 23, 2014

本文记录一下 TCP 的重传机制和拥塞处理.

TCP 用重传机制来保证所有的数据包都能到达

重传机制

举例, Client 发了 1,2,3,4,5 一共五份数据, Server 收到了1,2, 于是回 Ack=3, 接着收到了4, 此时3还没收到. 这种情况 TCP 会怎么办?

超时重传

Server 不回 Ack, 一直等3的到来, 一段时间后, Client 还没有收到3的 Ack, 于是 Client 进行重传, 对此 Client 有 2 种方式:

  • 只重传超时 timeout 的包, 比较慢
  • 重传超时 timeout 后的所有的包, 浪费带宽

2 种方式都要等到超时 timeout, 所以都不太好

快速重传机制 – Fast Retransmit

包没有连续到达, Server 就 Ack 最后那个可能丢了的包, 如果 Client 连续收到3次相同的Ack, 就重传. 不需要等超时 timeout 了再重传.

举例, Client 发了 1,2,3,4,5 一共五份数据, Server 收到了1,2, 于是回 Ack=3, 接着陆续收到了4,5, 还是 Ack=3, Client 收到了3个 Ack=3 的确认, 知道3还没有到, 于是马上重传2. 然后 Server 收到了3, 此时 3,4,5 都已经收到了, 于是 Ack=6.

但是此机制存在一个问题, Client 是重传3还是重传3,4,5呢? 因为 Client 并不清楚这连续的3个 Ack=3 是对应接收的哪个包传回来的? 也许 Client 发了10份数据, 是6,7,8传回来的呢. Client 很有可能要重传从2到10的数据, 很蛋疼.

SACK – Selective Acknowledgment

TCP 头里加一个 SACK, 用来记录哪些包到了, 哪些没有到. 解决 Fast Retransmit 算法遇到的问题.

滑动窗口 – Sliding Window

TCP 通过滑动窗口来解决数据包乱序的问题.

TCP 头部信息中有一个字段叫 Window(Advertised-Window), 是 Server 告诉 Client 自己还有多少缓冲区可以接收数据. 于是 Client 就根据此信息来发送数据, 不会导致 Server 处理不过来.

滑动窗口示意图:

  • Category #1: 已发送并收到 Ack 的数据
  • Category #2: 已发送还没有收到 Ack 的数据
  • Category #3: 未发送的数据, 但是数据已在缓冲区, 准备发送. Server 准备接收
  • Category #4: 未发送, 数据还未到缓冲区

滑动窗口的变化过程, 可以看一下油管中的视频演示(需要科学上网)

拥塞处理 – Congestion Handling

假设没有拥塞处理的 TCP, 在网络发生延迟时, TCP 会进行重传未收到 Ack 的数据, 加重网络负担, 延迟更加严重, TCP 又进行重传, 进入一个死循环.

TCP 的拥塞控制主要是四个算法: 慢启动, 拥塞避免, 拥塞发生, 快速恢复

慢启动

连接建立后, 慢慢提速, 过程如下:

cwnd(Congestion Window): 拥塞窗口 RTT(Round Trip Time): 一个数据包从发出到回来的时间 MSS(Max Segment Size): 最大报文段长度 ssthresh(slow start threshold): 慢启动阈值 RTO(Retransmission TimeOut): 重传超时

  1. 连接建好, 先初始化 cwnd = 1, 表明可以传一个 MSS 大小的数据。
  2. 每当收到一个 ACK, cwnd++, 线性上升
  3. 每当过了一个 RTT, cwnd = cwnd * 2, 指数让升
  4. 还有一个 ssthresh(slow start threshold)是一个上限，当 cwnd >= ssthresh 时, 就会进入"拥塞避免算法"

拥塞避免算法 – Congestion Avoidance

一般 ssthresh 的值是 65535 字节, 当 cwnd 达到这个值时后, 算法如下:

  • 收到一个 ACK 时, cwnd = cwnd + 1/cwnd
  • 当每过一个 RTT 时, cwnd = cwnd + 1

这样就可以避免增长过快导致网络拥塞, 慢慢的增加调整到网络的最佳值. 就跟在高速上开车差不多, 先在慢车道加速进入快车道, 时速120. 当发现前面车辆过多时(达到阻塞阈值), 刹车减速, 再等到车辆少时再加速.

快速恢复算法 – Fast Recovery

当 cwnd <= sshthresh 时, 出现丢包重传, sshthresh = cwnd / 2

快速恢复算法就是讲 cwnd 快速恢复到之前合理的大小, 算法如下:

  • cwnd = sshthresh + 3 * MSS （3的意思是确认有3个数据包被收到了）
  • 重传 Duplicated ACKs 指定的数据包
  • 如果再收到 duplicated Acks，那么 cwnd = cwnd +1
  • 如果收到了新的 ACK, 那么, cwnd = sshthresh, 然后就进入了拥塞避免的算法了

总结

看了网上一些 TCP 的 blog, 结合自己的理解, 汇总了下这些 blog 中提到有关 TCP 的内容, 也是比较重要的内容. 学到了不少, 同时也知道 TCP 复杂性, 还需要继续学习.
